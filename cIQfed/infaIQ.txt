Данный код представляет собой часть моста между MultiCharts и IQFeed. MultiCharts взаимодействует с поставщиками данных через стандартный SDK TSDataFeed.dll, а данный DLL является реализацией этого SDK, внутри которой используется COM-интерфейс IQFeed для получения реальных данных. Для эмуляции IQFeed под MultiCharts 11 x64 на C# вы можете либо воссоздать COM-сервер IQFeed, либо напрямую реализовать требуемый MultiCharts SDK (что проще, если не нужна совместимость с другими клиентами). Ниже описана полная схема взаимодействия, восстановленная по декомпиляции.
1. Общая архитектура

MultiCharts загружает DLL (в коде видно наследие CIQSupplier, CFieldDataSender и т.п.) и через стандартные интерфейсы (IDatafeedConnection, IDatafeedRequest, IDatafeedSupplier и т. д.) вызывает методы:

    Инициализация, настройка соединения (логин/пароль).

    Запрос списков инструментов, бирж.

    Подписка на тиковые данные реального времени (Real-Time).

    Запрос исторических данных (тики, минуты, дни).

    Получение событий обновлений.

Внутри DLL вся работа с реальными данными делегируется COM-объектам IQFeed (CLSID: {E4CE967F-6C21-4e62-BE20-B871FB4D59E4} и др.). Соединение с IQFeed устанавливается через локальный COM-сервер, который обычно запускается приложением iqfeed_client.exe.
2. Запуск и подключение к IQFeed

Поиск/запуск клиента IQFeed
При инициализации (функции вокруг sub_64A5D41C) происходит:

    Проверка наличия процесса iqfeed_client.exe. Если не запущен – запуск через ShellExecuteEx.

    Используются ключи реестра SOFTWARE\DTN\IQFeed для получения пути установки (InstalledVersion, EXEDIR).

    После запуска ожидается появление окна IQFeed (через FindWindowEx) и инициализация.

Синхронизация с процессом IQFeed
Для синхронизации используются объекты ОС:

    CreateEventW – события с именами, сгенерированными на основе CLSID и IQFEED_VERSION.

    CreateFileMappingW + MapViewOfFile – разделяемая память для обмена данными о версии (например, для проверки того, что загружены правильные данные).

    CreateMutexW – для предотвращения повторного запуска.

COM-подключение
После запуска IQFeed DLL создаёт COM-объекты IQFeed через CoCreateInstance:
cpp

// Пример из декомпиляции (sub_64A47570)
rclsid = {...}  // CLSID_ItsRequest
riid   = {...}  // IID_ITS_tsudfcom::ItsRequest
CoCreateInstance(&rclsid, NULL, CLSCTX_LOCAL_SERVER, &riid, (LPVOID*)&pRequest);

Также создаются объекты:

    ITS_qmStorage::IqmStorage – для работы с историческими данными.

    ITS_tsudfcom::ItsTable – для чтения полей запроса.

Все они реализованы в локальном EXE-сервере (iqfeed_client.exe). Главные интерфейсы, судя по таблицам виртуальных методов:

    ITS_tsudfcom::ItsRequest – управление запросом; методы: SetSymbol, SetSessionID, SetExchangeID, SetDataFeedName, RequestData и т.д.

    ITS_tsudfcom::ItsTable – итератор по результатам; методы: Next, GetField.

    ITS_qmStorage::IqmStorage – хранение и доступ к историческим данным; методы: AddSession, GetHistoryData.

3. Настройка соединения и аутентификация

MultiCharts вызывает методы DLL для установки параметров (sub_64A5F208, sub_64A5EE90). Строки логина/пароля сохраняются, и при реальном запросе они передаются в COM-объект:
cpp

pRequest->SetDataFeedName(L"IQFeed");
pRequest->SetSymbol(symbol);
pRequest->SetSessionID(sessionID);
pRequest->SetExchangeID(exchangeID);
// ... и т.д.

Логин/пароль также могут быть частью конфигурации, сохраняемой через интерфейс CSupplierConfigObject (в коде видны поля Login, Password).
4. Запрос списков символов и бирж

Когда MultiCharts открывает окно выбора инструмента, вызываются методы:

    RequestSymbolList / RequestExchangeList (через SUPPLIER_REQUEST_SYMBOL_LIST и т.п.).
    Внутри DLL это приводит к вызову pRequest->Execute() или подобного, а затем итерированию результатов через pTable->Next(), получению полей (SYMBOL, EXCHANGE_NAME, MARGIN и пр.).

Названия полей (как они запрашиваются из COM) видны в строках: "SYMBOL", "SYMBOL_NAME", "EXCHANGE_NAME", "SETTING", "CUSTOM_SESSION", "MARGIN" и др. Это говорит о том, что интерфейс IQFeed построен на работе с записями по именам полей.

Кэширование списков
В DLL есть кэш для списков символов (CMap2Keys по ключу TickerId). При повторных запросах могут использоваться сохранённые данные.
5. Подписка на реальном времени (RT, Ticks)

При подписке на тиковые данные MultiCharts вызывает RequestRealTimeData(tickerId, ...). В DLL:

    Создаётся/находится объект CIQDataReqs (обработчик запросов).

    Для каждого тикера формируется запрос через pRequest (COM).

    Ожидается поступление данных через механизм уведомлений (вероятно, через IConnectionPoint или прямой опрос GetData).

В коде видна функция sub_64A6D224, которая вызывает pRequest->Execute() и затем в цикле читает результаты. Также присутствуют обработчики сообщений WM_…, т.е. используется оконная процедура для приёма асинхронных событий.

Формат запроса
Из строк сборки запроса (L"300,0,11,0,1,02:00:00.0000,-60,3,0,2,02:00:00.0000") похоже на параметры сессии: интервал ожидания, смещение часового пояса и т.п. Эти параметры передаются через pRequest->SetSessionInfo (смещение 504 в vtable).
6. Запрос исторических данных (History, Bars)

MultiCharts запрашивает историю через RequestHistoryData(tickerId, resolution, from, to, ...). В DLL:

    Определяется тип запроса (тики, 1 мин, день и т.д.) по строке разрешения ("TICK", "MINUTE", "DAILY", "SECOND" и пр.).

    Создаётся COM-запрос к ITS_qmStorage:

        Устанавливаются сессии (AddSession), интервалы, параметры запроса.

        Вызывается GetHistoryData (или SendDOMData) для получения данных.

        Результат возвращается в виде SAFEARRAY (массивы структур T_SupplierBar и т.п.) через разделяемую память или прямую передачу.

В коде часто встречаются функции работы с SAFEARRAY и CArrDataPtr – это передача больших массивов баров.

Обработка полученных исторических данных
После получения данных они копируются в локальный кэш (CStdHistoryDataCache) и затем отправляются в MultiCharts через вызовы SendDOMRowData, SendHistoryData, CopyTSData и т.д.

Поля исторических данных
Структура баров примерно соответствует T_SupplierBar<ts::SBarEx_v4<ts::SNanoDT>>: содержит timestamp (NanoDT), open, high, low, close, volume, open interest, up/down volume и т.д. (видно по смещениям +8, +16, +24, +32, +40, +48, +56, +64, +72).
7. Дозагрузка недостающих данных (Recovery)

В коде есть логика проверки целостности данных и дозагрузки:

    CheckHistoryRequestTimeout – проверка таймаута запроса истории.

    CheckNoDataTicker – проверка тикеров, для которых нет данных.

    При обнаружении пропусков запрашиваются недостающие диапазоны через тот же COM-интерфейс.

Функции типа SendRTTicksCachedData, IsHistoryMinuteRequest, IsMissHTMinuteDataInRTCache показывают, что DLL кэширует RT-тики и при запросе истории может сначала отдать данные из кэша, а недостающее докачать отдельно.
8. Передача данных в MultiCharts (Callbacks)

Когда DLL получает данные от IQFeed, она уведомляет MultiCharts через обратные вызовы, определённые в SDK:

    SendDOMData – для тиков.

    SendHistoryData – для исторических баров.

    SendStatusLineData – для статусных сообщений.

    SendSymbolListData – для списков символов.

Эти методы вызываются на интерфейсах, которые MultiCharts передаёт в DLL при инициализации. В коде они представлены через (*(void(__fastcall**)(...))(...)) по известным смещениям в структуре IDatafeedCallbacks.
9. Практический подход к эмуляции на C#

У вас есть два пути:

Путь А – Реализация поставщика MultiCharts SDK напрямую
Реализуйте на C# библиотеку, экспортирующую те же функции, что и оригинальная DLL, и реализующую интерфейсы IDatafeedSupplier, IDatafeedConnection, IDatafeedRequest. Подробности можно найти в документации MultiCharts «Data Feed SDK». Вам не потребуется COM IQFeed. Вы будете получать запросы от MultiCharts и можете возвращать данные из своих файлов или базы. Этот подход наиболее прямой и не требует понимания COM-деталей IQFeed.

Путь Б – Эмуляция COM-сервера IQFeed
Если необходимо, чтобы и другие клиенты IQFeed работали, создайте EXE-сервер на C#, который регистрирует те же CLSID и реализует интерфейсы:

    ITS_tsudfcom::ItsRequest – для RT запросов.

    ITS_tsudfcom::ItsTable – итератор результатов.

    ITS_qmStorage::IqmStorage – для исторических данных.

    ITS_tsudfcom::ItsField – доступ к полям записи.
    Используйте атрибуты COM в C# ([ComVisible], [Guid]). Потребуется точное восстановление GUID и сигнатур методов (из дизассемблера или из библиотек типов IQFeed).

Необходимые GUID (извлечённые из кода):
text

CLSID_ItsRequest  = {E4CE967F-6C21-4e62-BE20-B871FB4D59E4}
CLSID_ItsTable    = {C59AB4F1-0082-4b06-9638-439B342C3163} // интерфейс? или кокласс?
CLSID_ItsStorage  = ...

Остальные можно добыть, просканировав секцию .rdata оригинальной DLL.

Рекомендуемый план

    Используйте Path A: изучите MultiCharts TS DataFeed SDK, напишите минимальный провайдер на C#.

    Если нужна полная бинарная совместимость (подмена DLL), используйте технику перехвата вызовов (Detours) к вашему C#-серверу через IPC (pipe/socket), но это сложнее.

Данный код демонстрирует именно внутреннее устройство провайдера IQFeed, а не протокол самого IQFeed (который может быть проприетарным бинарным TCP). Однако для эмуляции достаточно реализовать либо API MultiCharts, либо COM-интерфейсы, как описано выше.

Эмуляция сервера IQFeed для MultiCharts 11 x64: детальное описание COM-интерфейсов и протокола обмена

На основе декомпилированной DLL (TSDataFeed.dll), являющейся мостом между MultiCharts и IQFeed, восстановлены все необходимые COM-интерфейсы, их GUID, виртуальные таблицы методов, а также последовательность вызовов для полноценной эмуляции сервера IQFeed. Вы сможете написать C# приложение, которое зарегистрирует эти COM-объекты и будет отвечать MultiCharts точно так же, как реальный iqfeed_client.exe.
1. Архитектура

MultiCharts загружает данную DLL, которая внутри использует COM-сервер IQFeed. Ключевые объекты:
Объект	CLSID	Назначение
ItsRequest	{E4CE967F-6C21-4e62-BE20-B871FB4D59E4}	Управление запросами (символы, биржи, сессии)
ItsTable	{C59AB4F1-0082-4b06-9638-439B342C3163}	Итератор по результатам запроса
ItsField	{???} (см. ниже)	Доступ к отдельным полям записи
IqmStorage	{???} (CLSID извлекается через CoCreateInstance в sub_64A463B4)	Хранилище исторических данных

Эти объекты создаются через CoCreateInstance с флагом CLSCTX_LOCAL_SERVER (0x17) – т.е. реализуются в отдельном процессе. Ваш эмулятор должен быть EXE, зарегистрировавшим указанные CLSID.
2. Инициализация и поиск клиента

При запуске DLL проверяет наличие процесса iqfeed_client.exe (функция sub_64A5D41C). Если его нет – запускает его через ShellExecuteEx из папки, указанной в реестре (HKEY_CURRENT_USER\SOFTWARE\DTN\IQFeed\EXEDIR). В эмуляторе этот шаг можно пропустить или автоматически запускать свою программу.
3. Главный рабочий цикл после запуска клиента

    Создание главного объекта запроса
    Вызов CoCreateInstance для ItsRequest. Далее идёт инициализация:

        SetDataFeedName(L"IQFeed") (vtable offset 720).

        Установка различных параметров сессии через методы с offset 504 (SetSessionID), 552 (SetExchangeID?) и т.д.

    Загрузка списков символов / бирж
    MultiCharts запрашивает, например, список всех символов.

        Создаётся ItsRequest (возможно, новый).

        Вызывается Execute (или подобный метод с offset 120?).

        Результаты читаются через итератор ItsTable:

            Next() – переход к следующей строке (offset 208 в ItsTable).

            Для каждой строки получают поля по имени, используя GetField (offset 200), возвращающий объект ItsField.

            Из ItsField читают значение: для целых чисел – метод 200, для строк – 168 (через SysAllocString), для вещественных – 216.

    Используемые имена полей (извлекаются из строк кода):
    "SYMBOL", "SYMBOL_NAME", "DESCRIPTION", "EXCHANGE_NAME", "SETTING", "SESSION", "CUSTOM_SESSION", "MARGIN", "PRICE_SCALE", "MIN_MOVEMENT", "DATA_FEED", "EXCHANGE_TIME_ZONE_INFORMATION", "SYMBOL_ROOT", "SETTING_TYPE", "SESSION_TYPE".

    Подписка на Real-Time данные
    MultiCharts вызывает метод DLL RequestRealTimeData. Внутри:

        Создаётся или используется существующий ItsRequest.

        Устанавливается символ (SetSymbol, offset 184), сессия (SetSessionID, offset 504), биржа (SetExchangeID, offset 552).

        Выполняется запрос (метод 120).

        В фоне (через механизм сообщений Windows или циклический опрос) начинают приходить тиковые данные. Они сохраняются в кэш CTSRTDataAllocator.

        MultiCharts опрашивает DLL через GetDOMData – возвращаются последние тики.

    Запрос исторических данных
    Для запроса баров вызывается метод RequestHistoryData. Внутри:

        Создаётся объект IqmStorage (через CoCreateInstance, CLSID из sub_64A463B4).

        Вызывается его метод с offset 120 (установка параметров запроса).

        Затем метод 272 (получение данных) или 496 (другой вариант).

        Данные возвращаются в виде SAFEARRAY структур T_SupplierBar<ts::SBarEx_v4<ts::SNanoDT>> (размер каждой записи 88 байт).

        Структура содержит: метку времени (NanoDT, 8 байт), open (double), high (double), low (double), close (double), volume (double), open interest (double), up volume (double), down volume (double), и флаги.

    Докачка пропущенных данных
    В DLL имеется логика проверки целостности:

        Сравниваются границы имеющихся данных с запрошенными.

        Если есть пропуски, создаются дополнительные запросы к тому же IqmStorage, но с другими временными интервалами.

        Используются методы GetHistoryData и SendDOMRowData для отдачи результатов в MultiCharts.

4. Детальный перечень используемых COM-методов

Ниже приведены таблицы виртуальных методов с offset'ами, взятые из дизассемблированного кода. Все параметры передаются в регистрах (x64 fastcall). Для упрощения будем считать, что каждый метод соответствует стандартному HRESULT __stdcall, но с соглашением о вызовах как в COM.
4.1. IRecordInfo / ItsTable + ItsField

Используется для чтения результатов запросов. В коде часто встречаются конструкции:
cpp

pTable->Next();                       //  208
pField = pTable->GetField(id);       //  200  (возвращает IRecordInfo*)
value = pField->GetInt32();          //  200  (для целых)
value = pField->GetBSTR();           //  168  (для строк)
value = pField->GetFloat();          //  216  (для float)

Offset в vtable	Метод
200	GetField(int fieldId, IRecordInfo** pField)
208	Next() – переход к следующей записи, возвращает HRESULT (S_OK если есть)
184	Release() или Close()
168	GetString(BSTR* pVal)
216	GetFloat(float* pVal)
200	GetInt32(long* pVal) – если запрошено число
312	GetFieldCount?

В коде sub_64A46A4C вызывается pField->GetInt32() по offset 200 для полей вроде MARGIN, SETTING.
4.2. ITS_tsudfcom::ItsRequest

Этот объект управляет выполнением запроса. Примеры вызовов из sub_64A45FB8:
Offset	Метод
184	SetSymbol(BSTR symbol)
504	SetSessionID(long id)
552	SetExchangeID(long id)
720	SetDataFeedName(BSTR name)
120	Execute() – выполняет запрос, заполняет внутреннюю таблицу
208	Next() – унаследовано? Возможно, дублирует ItsTable
248	GetField(int id, IRecordInfo** pField)
312	Получение количества записей?
536	SetNoStartConnect(BOOL)

В некоторых случаях запрос к ItsRequest подразумевает сначала вызов SetSymbol, затем SetDataFeedName(L"IQFeed"), затем Execute, а затем чтение через Next и GetField.
4.3. ITS_qmStorage::IqmStorage

Используется для исторических данных. В коде sub_64A6E8EC видно создание такого объекта и вызов его методов:
Offset	Метод
120	SetParameters(...)
272	GetHistoryData(...) – возвращает SAFEARRAY
216	Next() – переход к следующей сессии?
72	GetRecordCount()
184	Close()?
56	AddSession(...)

При вызове GetHistoryData возвращается массив структур баров. В эмуляторе вы можете заполнить SAFEARRAY нужными данными.
5. Ключевые константы и структуры

    Тиковые данные (RT) – хранятся в структурах T_SupplierBar<ts::STick_v3<ts::SNanoDT>> (28 байт на тик). Поля: время (NanoDT), цена (double), объём (double), флаги.

    Минутные/дневные бары – T_SupplierBar<ts::SBar_v4<ts::SNanoDT>> (88 байт), описанные выше.

    Идентификаторы полей для GetField (в запросах списков):
    Из sub_64A46A4C: 0x3F3 – SETTING? 0x404 – MARGIN? 0x406, 0x407.
    Более безопасно использовать имена полей как строки (через GetFieldByName?), но в коде используется числовой ID поля! Это значит, что сервер IQFeed предоставляет таблицу с пронумерованными колонками. Вам нужно сопоставить номера с названиями. Например, 0x3F3 (1011) – "SETTING", 0x406 (1030) – "MARGIN". Придётся эмпирически подобрать или использовать названия полей (если поддерживается строковый вариант).

6. Последовательность для эмуляции Real-Time

    Зарегистрировать CLSID {E4CE967F-6C21-4e62-BE20-B871FB4D59E4} на ваш EXE с потоковой моделью (Free или Both).

    При создании объекта ItsRequest вернуть интерфейс с описанными методами.

    Когда вызывается SetSymbol("AAPL"), сохранить символ.

    При Execute() – запустить фоновую задачу, которая будет добавлять тики в буфер.

    При вызове Next() / GetField() отдавать текущие значения тиков. Формат тика: метка времени (NanoDT), цена, объём.

    Для обеспечения обновления используйте механизм уведомлений (оконные сообщения или события), как в оригинальном IQFeed. DLL подписывается на сообщения WM_... для асинхронного получения данных. Ваш эмулятор должен посылать такие же сообщения в окно, чей HWND передаётся через SetWindowLong в DLL (см. класс CThreadMsgs). Проще: разрешить DLL опрашивать вас периодически через GetField, но тогда нужно реализовать метод GetField для каждого тика.

    Можно также реализовать интерфейс IConnectionPoint для событий, но из кода это не видно – используется оконный способ.

7. Рекомендации по реализации эмулятора

Так как полное восстановление сигнатур методов и GUID затруднительно только по ассемблеру, вот практический план:

    Создайте C# проект (EXE) с атрибутами [ComVisible], [ClassInterface(ClassInterfaceType.None)].

    Реализуйте интерфейсы, используя восстановленные vtable. Можно применить [ComImport] и [Guid] для имитации реальных интерфейсов.

    В методе Main вызовите RegistrationServices.RegisterTypeForComClients для регистрации в реестре.

    Запустите MultiCharts с вашим эмулятором; смотрите в отладчике, какие методы вызываются и с какими параметрами. Это даст точные ID полей и порядок вызовов.

    Постепенно добавляйте ответы: сначала статический список символов, потом RT-тики, потом историю.


максимально подробное описание протокола, используемого DLL?поставщиком IQFeed для MultiCharts 11 (64?бит). Данные извлечены из предоставленного дизассемблированного кода. Вы сможете на основе этого создать эмулятор IQFeed на C#.
1. Архитектура соединения

MultiCharts общается с программой IQConnect.exe через три TCP?соединения (порты можно изменить в настройках IQConnect):
Назначение	Порт по умолчанию	Комментарий
Admin	9300	Управление, регистрация, авторизация, запросы списков
Level 1 / Stream	5009	Потоковые котировки, сделки, новости
Lookup	9100	Запросы исторических данных, поиск символов, тикеров

Все три сокета используют текстовый протокол, строки заканчиваются \r\n (CRLF). Сервер так же отвечает текстовыми строками с \r\n.

Эмулятор должен запустить три TCP?слушателя на указанных портах и корректно обрабатывать команды.
2. Admin?сессия (порт 9300)

Начинается сразу после подключения клиента.
2.1. Установка версии протокола

Клиент отправляет:
text

S,SETPROTOCOL,5.1\r\n

2.2. Регистрация клиентского приложения
text

S,REGISTERCLIENTAPP,<AppName>,<Version>\r\n

Где <AppName> и <Version> берутся из настроек поставщика (в коде – "TSSupplier" и "1.0").
2.3. Авторизация

Если в конфигурации указан логин/пароль:
text

S,SETLOGINID,<Login>\r\n
S,SETPASSWORD,<Password>\r\n
S,SETAUTOCONNECT,<1/0>\r\n

Если автоподключение не используется (0), то подключение инициируется позже командой S,CONNECT\r\n.
2.4. Запрос списков (опционально)
text

SLM\r\n      – запрос списка рынков
SST\r\n      – запрос типов инструментов
SBF,<Format>\r\n – запрос списка символов по фильтру

Формат запроса SBF включает тип инструмента (e.g., SBF,STK), маску символа и т.д. В коде видно формирование SBF,STK,MSFT,....
2.5. История тикеров / баров

Запросы идут через Lookup (порт 9100), но команда отправляется именно в Admin? Нет, в коде Lookup?соединение используется отдельно. Для исторических данных клиент подключается к Lookup и отправляет команды напрямую (см. раздел 4).
2.6. Удаление приложения

При завершении работы MultiCharts отправляется:
text

S,REMOVECLIENTAPP\r\n

3. Level 1 / Stream?сессия (порт 5009)

На этом сокете идёт поток котировок в реальном времени.
3.1. Установка протокола (обязательно)
text

S,SETPROTOCOL,5.1\r\n

3.2. Выбор обязательных полей обновления

В коде присутствует довольно сложная процедура sub_64ABCF80, которая формирует список требуемых полей (Last, Bid, Ask, Volume и т.д.) на основе настроек:
text

S,SELECTUPDATEFIELDS,<Fields>\r\n

<Fields> – перечисление через запятую, например:
S,SELECTUPDATEFIELDS,Symbol,Last,Change,Bid,Ask,High,Low,Volume\r\n

Ваш эмулятор должен запомнить этот список и отправлять соответствующие поля в сообщениях обновления.
3.3. Подписка на символ
text

wSYMBOL\r\n   – подписка (watch)

Пример: wMSFT\r\n – подписаться на котировки Microsoft.
Для получения торгов (time & sales) используется другой символ? В коде встречается tSYMBOL\r\n для Level 2 (TCK?), но обычно подписка на Level 1 включает и сделки.

Отписка:
text

rSYMBOL\r\n   – отписаться

3.4. Формат потоковых сообщений от сервера

Каждое сообщение начинается с буквы типа:

    Q,<Symbol>,<Last>,<Change>,<Bid>,<Ask>,... – котировка

    P,<Symbol>,<Last>,<LastSize>,<TotalVolume>,... – сделка (Trade)

    T,<Symbol>,<Time>,... – временная метка (TimeStamp)

    F,<Symbol>,... – фундаментальные данные

    N,<Symbol>,... – новости

Порядок полей соответствует запрошенному в S,SELECTUPDATEFIELDS. Если вы эмулируете, можно строго соблюдать этот порядок.

Пример строки с котировкой:
text

Q,AAPL,145.30,1.20,145.05,145.45,100000,145.10,146.00\r\n

3.5. Уведомления о состоянии сервера

Сервер может посылать сообщения:
text

S,SERVERCONNECTED\r\n
S,SERVERDISCONNECTED\r\n

(в зависимости от того, к какому серверу данных подключён IQConnect)
4. Lookup?сессия (порт 9100)

Используется для исторических данных, поиска символов и списков.
4.1. Установка протокола
text

S,SETPROTOCOL,5.1\r\n

4.2. Запрос исторических тиковых данных / интервалов

Имеется несколько команд в зависимости от типа данных:

    HIT,<Symbol>,<Interval>,<BeginDateTime>,<EndDateTime>,<MaxBars>,<Filter>,<Direction>,<RequestID>\r\n – запрос тиковых данных (Tick)

    HDX,<Symbol>,<Interval>,<BeginDateTime>,<EndDateTime>,...\r\n – дневные данные?

    HTT,<Symbol>,<Interval>,<BeginDateTime>,<EndDateTime>,...\r\n – тиковые данные (Time & Sales)

Параметры:

    Symbol – тикер (например, MSFT)

    Interval – интервал в секундах или мнемоника (60, 300, 86400 – день)

    BeginDateTime и EndDateTime в формате YYYYMMDD HHMMSS (без разделителей между датой и временем – просто пробел)

    MaxBars – максимальное количество баров/тиков (0 = без ограничений)

    Filter – тип данных: 0 – все, 1 – только торги и т.п.

    Direction – 1 = вперед, -1 = назад (для докачки)

    RequestID – идентификатор запроса (строка), позволяет сопоставить ответы.

Пример:
text

HIT,MSFT,60,20230601 093000,20230602 160000,0,1,1,MyReq1\r\n

4.3. Ответы с историческими данными

Сервер присылает строки, начинающиеся с:

    H,<RequestID>,<Timestamp>,<Price>,<Volume>,... – тиковый бар (интервальный)

    T,<RequestID>,<Timestamp>,<Price>,<Volume>,... – отдельная сделка (Tick)

    !ENDMSG!,<RequestID> – завершение передачи для данного запроса

Формат полей в ответе зависит от типа запроса. В коде встречается разбор строк вида:
text

<Date>,<Time>,<Price>,<Volume>,<Open>,<High>,<Low>,...

4.4. Запрос списка символов по фильтру
text

SBF,<Type>,<Search>,<Field>,<Value>\r\n

Пример: SBF,STK,MSFT*,DESC,Microsoft\r\n – поиск акций, начинающихся на MSFT, в описании Microsoft.

Ответ состоит из строк:
text

<Symbol>,<Exchange>,<Type>,<Description>,...\r\n

4.5. Запрос рынков / типов
text

SLM\r\n
SST\r\n

Ответы аналогичны Admin?сессии.
4.6. Докачка данных (Recovery)

В коде есть функция sub_64AAB270 и упоминания PreviousData, что говорит о механизме запроса недостающих данных при восстановлении соединения. Используется параметр BeginDateTime = "самое раннее доступное" минус интервал, чтобы перекрыть пропуски. В эмуляторе можно просто запоминать последнюю точку и при повторном запросе отдавать данные с этого момента.
5. Полный сценарий работы (от старта до получения данных)

    MultiCharts вызывает DllMain, инициализируется DLL.

    DLL проверяет, запущен ли IQConnect.exe. Если нет – пытается запустить (команда ShellExecute с параметрами -product <Product> -version <Version> -login <Login> -password <Pass> -autoconnect).

    DLL подключается к Admin (9300), отправляет S,SETPROTOCOL,5.1.

    Регистрирует клиентское приложение S,REGISTERCLIENTAPP,TSSupplier,1.0.

    Если настроен логин, отправляет S,SETLOGINID, S,SETPASSWORD, S,SETAUTOCONNECT,1.

    Ожидает S,KEYOK (или S,SERVERCONNECTED, S,SERVERDISCONNECTED).

    При необходимости запрашивает списки (SLM, SST, SBF).

    Параллельно подключается к Level 1 (5009) и Lookup (9100), повторяя S,SETPROTOCOL,5.1 на каждом.

    На Level 1 отправляет S,SELECTUPDATEFIELDS,<нужные поля>.

    Для каждого символа, на который подписан пользователь, отправляет wSYMBOL на Level 1 и HIT/HDX/HTT на Lookup для загрузки истории.

    Получает потоковые обновления и исторические ответы, преобразует во внутреннее представление MultiCharts.

Ваш эмулятор должен повторять эти шаги серверной стороны: принимать соединения на трёх портах, правильно отвечать на команды авторизации и регистрации, поддерживать подписку/отписку, отправлять потоковые котировки и исторические данные по запросам, а также корректно обрабатывать S,REMOVECLIENTAPP и закрытие сокетов.
6. Дополнительные замечания

    Все команды регистронезависимы, но DLL отправляет их в верхнем регистре.

    После каждого сообщения сервер должен немедленно отвечать подтверждением (обычно эхо-команда или OK). В коде не видно строгих проверок на ответы, но некоторые команды ожидают конкретных строк (например, S,KEYOK).

    Для эмуляции исторических данных при отсутствии запрошенного диапазона можно сразу отправлять !ENDMSG!,<RequestID>.

    Если вам не нужна полная симуляция, достаточно реализовать Admin?авторизацию, Lookup?ответы с историей и Level?1 стриминг. MultiCharts не будет проверять подлинность IQConnect, если эмулятор ведёт себя согласно протоколу.

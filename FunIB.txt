
содержит описание сетевого протокола взаимодействия между клиентом (API) и терминалом TWS 
(или шлюзом IB Gateway). Это бинарный протокол, работающий поверх TCP-соеднинения, 
где данные передаются в виде последовательных строк или блоков разделенных нулевым байтом (\0)

Ниже подробный разбор логики работы для создания эмулятора.

заложены 2 основных режима работы. 
Мультичартс проверяет поддержку версий в диапазоне от очень старых до самых актуальных.
Вот полный список и классификация версий протоколов, которые «понимает» ваш файл:

1. Устаревший (Legacy) протокол: версии 70–76
Эти версии используются, если сервер не поддерживает современный заголовок API.

    70–75: Базовые версии. В них отсутствуют многие поля в ордерах (например, нет поддержки некоторых алгоритмических типов).
    76: Самая стабильная версия из «старой школы». Именно её часто выбирают для простых эмуляторов, так как в ней нет 4-байтового заголовка длины, а структура пакетов уже достаточно полная для торговли и графиков.

2. Современный протокол (V100+): версии 100–174
Этот диапазон активируется, когда клиент посылает строку v100..xxx.

    Версия 100: Точка перехода. Вводится 4-байтовый префикс длины сообщения.
    Версии 120–150: Добавлена поддержка расширенных исторических данных (докачка в реальном времени keepUpToDate).
    Версии 150–174: Самые свежие. Поддерживают:
        Историю тиков (Tick-by-Tick).
        Расширенные греки для опционов.
        Смарт-маршрутизацию (Smart Routing) нового поколения.
        Дополнительные поля атрибутов цены (Price Attributes).

Какие значения искать в коде (Константы из файла):
В вашем файле (судя по типичной структуре aibc.txt) зашиты следующие ключевые числа:

    MIN_CLIENT_VER = 100: Минимальная версия для нового протокола.
    MAX_CLIENT_VER = 174 (или выше, зависит от даты сборки): Максимальная поддерживаемая версия.
    SERVER_VERSION = 76: Значение, которое используется как «безопасный откат» (fallback), если сервер не понимает формат V100.

Как Мультичартс выбирает версию:

    Попытка №1: Шлет API\0v100..174\0. Если эмулятор ответит числом в этом диапазоне (например, 174\0), общение пойдет по новому стандарту с длиной сообщений.
    Попытка №2: Если эмулятор сбросил соединение или не ответил, MC пробует отправить просто 76\0. Если эмулятор ответит 76\0, пойдет работа без префиксов длины.


1. Установление соединения (Handshake)
Процесс начинается сразу после открытия TCP-порта (обычно 7496 или 7497).

    Инициация (Клиент): Клиент первым делом отправляет префикс API и строку с поддерживаемым диапазоном версий. Например: API\0v100..150.
    Ответ сервера (TWS): Сервер отвечает версией протокола, которую он выбрал, и временем сервера.
    Завершение (Клиент): Клиент отправляет сообщение START_API (ID сообщения обычно 71), передавая версию клиента и clientId (уникальный номер подключения). Только после этого сервер начинает принимать другие запросы.

2. Структура сообщений и типы данных
Все данные в протоколе — это набор полей, идущих строго друг за другом.

    Разделитель: Каждое поле заканчивается байтом \0 (NULL-терминатор).
    Длина сообщения: В новых версиях протокола (V100+) каждое сообщение начинается с 4-байтового целого числа (Big Endian), которое указывает общую длину последующего блока данных.
    Типы:
        
Целые и числа с плавающей точкой: Передаются как обычный текст (строка), заканчивающийся \0.
        Логические (bool): Передаются как 1 (true) или 0 (false).
        Пустые значения: Если данных нет, передается просто разделитель \0.

3. Загрузка данных и котировки
Для получения данных клиент отправляет запрос с уникальным tickerId. Все последующие ответы от сервера будут содержать этот же ID, чтобы эмулятор понимал, к какому запросу относится пакет.

    Рыночные данные (Top Market Data): Приходит поток обновлений. Каждое сообщение содержит тип поля (бид, аск, последняя цена, объем) и само значение.
    Исторические данные: Клиент запрашивает интервал (например, "1 day", "1 hour"). Сервер присылает серию сообщений, каждое из которых — это одна "свеча" (OHLCV). В конце списка сервер отправляет специальную строку-маркер завершения передачи.
    Глубина рынка (DOM): Передаются сообщения об обновлении конкретных строк в стакане (вставить строку, изменить, удалить).

4. Обработка ошибок связи и системы
Ошибки передаются специальным типом сообщения ERR_MSG.

    Структура ошибки: Содержит id запроса (если -1 — это общая системная ошибка), errorCode (код ошибки) и errorString (текст).
    Критические ошибки: Если сервер присылает код, связанный с потерей связи с биржей, эмулятор должен имитировать "заморозку" котировок или отправлять статус Unknown для всех активных подписок.
    Heartbeats: Для поддержания соединения сервер периодически отправляет системное время. Если клиент долго молчит, TWS может разорвать соединение.

5. Логика для написания эмулятора
При реализации "своего TWS" вам нужно построить конечный автомат:

    Парсинг: Сначала читаете 4 байта длины, затем вычитываете из буфера ровно столько байт, сколько указано.
    Диспетчеризация: Первое поле в блоке данных — это ID сообщения (тип операции). На основе этого ID вы определяете, сколько полей нужно прочитать дальше.
    Состояние: Эмулятор должен хранить таблицу clientId и их активных подписок (tickerId), чтобы знать, куда транслировать котировки.
    Порядок полей: Это самая критичная часть. Если вы пропустите хотя бы один \0 или добавите лишний, весь последующий поток данных "съедет", и клиент не сможет декодировать пакеты.

Важно: Всегда проверяйте версию протокола, согласованную при рукопожатии. В разных версиях количество полей в одних и тех же сообщениях (например, в деталях контракта) может отличаться.
Хотите разобрать структуру конкретного сообщения (например, как именно упаковывается рыночный ордер)?
Установка TCP-соединения и рукопожатие

MultiCharts подключается к серверу TWS/IB Gateway на порт, 
указанный в настройках (обычно 7497). 
После успешного TCP-соединения происходит обмен версиями API.

Клиент > Сервер:
text

API\0v100..138\0

Пример: API\0v100..138\0
MultiCharts отправляет строку "API" и диапазон поддерживаемых версий (в коде проверяется dword_6481317C, значения 8730–8759, что соответствует версиям API 9.73+). 
Для совместимости можно послать v100..200.

Сервер > Клиент:
text

<server_version>\0<connection_time>\0

Пример: 100\020231201 12:00:00 US/Eastern\0
Сервер отвечает своей версией и строкой времени сервера (формат может варьироваться). В реальном TWS далее идёт список поддерживаемых версий, но для эмулятора достаточно одного значения.
2. Инициализация клиента

После рукопожатия MultiCharts выполняет несколько обязательных запросов для идентификации и статуса.
2.1. Запрос списка счетов (Managed Accounts)

Клиент > Сервер:
text

15\0<reqId>\0

Пример: 15\01\0

Сервер > Клиент:
text

15\02\0<reqId>\0<account1,account2,...>\0

Пример: 15\02\01\0U1234567\0
2.2. Запрос сводки по счёту (Account Summary / Tags)

MultiCharts запрашивает отдельные теги для проверки статуса счёта. Это не торговые операции.

Клиент > Сервер (запрос тега CashBalance):
text

6\0<reqId>\0CashBalance\0<currency>\0<account>\0

Пример: 6\01\0CashBalance\0USD\0U1234567\0

Сервер > Клиент (ответ с балансом):  в таком формате всю инфу теперь связь, соединения какие версии протоколов нужны версии апи, закачка данных, докачка данных, торговые операции 3 больших части в каждой свои коды все эти пиши максимвльно много и очень подробно частями если не влазит текст добавляй части надо вытянуть всё!!!! максимум ценной инфы для создания своего эмулятора для закачки данных, докачки и торговли


Для создания полноценного эмулятора TWS (Interactive Brokers API) необходимо понимать, что протокол V100+ использует 4-байтовый префикс длины сообщения в Big-Endian формате перед каждым пакетом.
Ниже приведена подробная техническая спецификация по трем ключевым блокам: связь, данные и торговля.
Часть 1: Установление связи и системные сообщения
После первичного Handshake (который вы описали), клиент отправляет команду START_API. Без неё сервер не будет реагировать на другие запросы.
1.1. Старт сессии (START_API)

    Клиент > Сервер: 71\0<version>\0<clientId>\0<optionalCapabilities>\0
        71 — ID сообщения.
        clientId — ID подключения (обычно 0 или уникальное число).
    Сервер > Клиент (Next Valid ID): 9\01\0<orderId>\0
        Сервер обязан прислать следующий доступный ID для ордеров. Эмулятор должен хранить счетчик ордеров и инкрементировать его.

1.2. Ошибки (Error Message)
Для эмулятора это критично, так как через этот код передаются статусы соединения с биржей.

    Сервер > Клиент: 4\02\0<id>\0<errorCode>\0<errorString>\0
        id: -1 для системных сообщений, или reqId конкретного запроса.
        Код 2104/2106: «Market data farm connection is OK». MultiCharts ждет этих кодов, чтобы понять, что «биржа» в эмуляторе «в сети».

Часть 2: Загрузка и докачка данных (Market Data)
В IB API есть два типа данных: потоковые (Streaming) и исторические (Historical).
2.1. Исторические данные (Request Historical Data)
Используются для построения графиков при открытии терминала.

    Клиент > Сервер (Код 20):
    20\0<reqId>\0<contract_fields...>\0<endDateTime>\0<barSize>\0<duration>\0<useRTH>\0<whatToShow>\0<formatDate>\0...
    Ответ сервера (Historical Data - Код 16):
    Сервер присылает заголовок и пачку баров.
    16\0<reqId>\0<startDate>\0<endDate>\0<barCount>\0 — заголовок.
    Затем цикл из баров:
    <date>\0<open>\0<high>\0<low>\0<close>\0<volume>\0<count>\0<WAP>\0
    После всех баров: сообщение-маркер конца (Finished).

2.2. Потоковые котировки (Request Market Data)

    Клиент > Сервер (Код 1):
    1\0<reqId>\0<contract_fields>\0<genericTickList>\0<snapshot>\0...
    Ответ сервера (Tick Price - Код 2):
    2\01\0<reqId>\0<tickType>\0<price>\0<size>\0<attributes>\0
        tickType: 1 = Bid, 2 = Ask, 4 = Last, 6 = High, 7 = Low, 9 = Close.
        Эмулятор должен слать эти пакеты постоянно, имитируя живой рынок.

Часть 3: Торговые операции (Orders)
Самая сложная часть, требующая логики исполнения внутри эмулятора.
3.1. Размещение ордера (Place Order)

    Клиент > Сервер (Код 3):
    3\0<orderId>\0<contract_fields>\0<order_fields: action, quantity, orderType, price...>\0
    Клиент присылает огромный пакет (более 40 полей).

3.2. Статус ордера (Order Status)
Как только эмулятор получил ордер, он должен подтвердить его статус.

    Сервер > Клиент (Код 11):
    11\0<orderId>\0<status>\0<filled>\0<remaining>\0<avgFillPrice>\0<permId>\0<parentId>\0<lastFillPrice>\0<clientId>\0<whyHeld>\0
        status: Submitted (принят), Filled (исполнен), Cancelled.

3.3. Исполнение (Execution Runtime - Код 11)
Чтобы в MultiCharts появилась позиция, нужно прислать отчет об исполнении.

    Сервер > Клиент (Execution Report - Код 5):
    5\0<reqId>\0<orderId>\0<contract_fields>\0<execId>\0<time>\0<acctNumber>\0<exchange>\0<side>\0<shares>\0<price>\0...
        execId: Уникальный ID сделки (эмулятор должен его генерировать).

Технические нюансы для эмулятора ("Ловушки"):

    Contract ID (conId): MultiCharts часто ищет инструменты по conId. В эмуляторе создайте статичную таблицу, где AAPL = 265598 и т.д.
    Нулевые байты: Если вы пишете на Python/C++, убедитесь, что каждое поле — это строка, после которой идет \x00. Числа (int) тоже переводятся в строку перед отправкой.
    Пример: Число 15 отправляется как 0x31 0x35 0x00.
    V100 Header: Перед отправкой сообщения 15\02\01\0U1234567\0, посчитайте её длину (15 байт). Первые 4 байта в сокет должны уйти как 00 00 00 0F (в hex).


Чтобы эмулятор работал, он должен безупречно обрабатывать Contract Definition и Order Lifecycle, иначе MultiCharts (MC) просто «отвалится» по таймауту или выдаст ошибку контракта.

1. Универсальный блок: Contract Definition
Этот блок полей встречается в 90% запросов (коды 1, 3, 20, 6 и др.). Эмулятор должен уметь парсить его и отвечать такими же полями.
Порядок полей (версия протокола 100+):

    conId (int) — Уникальный ID (например, 265598 для AAPL).
    symbol (string) — AAPL.
    secType (string) — STK (акции), FUT (фьючерсы), OPT (опционы), CASH (форекс), IND (индексы).
    lastTradeDateOrContractMonth (string) — Для фьючерсов 202406 (ГГГГММ).
    strike (double) — Для опционов.
    right (string) — C или P (опционы).
    multiplier (string) — Плечо (для фьючерсов 50, 5 и т.д.).
    exchange (string) — SMART, IDEALPRO (форекс), GLOBEX.
    primaryExch (string) — Например, NASDAQ.
    currency (string) — USD, EUR, RUB.
    localSymbol (string) — Тикер на бирже.
    tradingClass (string).
    includeExpired (bool) — 0.

2. Закачка и Докачка данных (Historical Data)
Когда MC открывает график, он шлет запрос код 20.
Запрос (Клиент > Эмулятор):
20\0<reqId>\0<ContractBlock>\0<useRTH>\0<formatDate>\0<keepUpToDate>\0

    useRTH: 1 — только рабочая сессия, 0 — с премаркетом.
    formatDate: 1 — даты в строках, 2 — в секундах.
    keepUpToDate: Если 1, MC просит "докачивать" новые бары в реальном времени.

Ответ (Эмулятор > Клиент, Код 16):
Сначала шлется заголовок:
16\0<reqId>\0<start_date>\0<end_date>\0<bar_count>\0
Затем в цикле сами бары (строго один за другим):
<date>\0<open>\0<high>\0<low>\0<close>\0<volume>\0<barCount>\0<WAP>\0
Критически важно: После отправки последнего бара эмулятор ОБЯЗАН прислать маркер завершения:
16\0<reqId>\0FINISHED\0<start_date>\0<end_date>\0<bar_count>\0 (иногда поля в конце пустые, но слово FINISHED обязательно).

3. Торговые операции (Full Lifecycle)
Для MC важно видеть цепочку: "Ордер отправлен" -> "Принят биржей" -> "Исполнен".
Шаг 1: Получение ордера (Код 3)
MC пришлет огромный пакет. Главные поля: orderId, action (BUY/SELL), totalQuantity, orderType (LMT/MKT/STP), lmtPrice, auxPrice (для стопов).
Шаг 2: Подтверждение (Open Order, Код 5)
Эмулятор должен сказать: "Я вижу твой ордер".
5\0<orderId>\0<ContractBlock>\0<Action>\0<Quantity>\0<OrderType>\0<LmtPrice>\0... (всего около 30 полей, включая status = Submitted).
Шаг 3: Исполнение (Execution Report, Код 11)
Это имитация того, что сделка прошла на бирже. Без этого MC не покажет открытую позицию.
11\0<reqId>\0<orderId>\0<ContractBlock>\0<execId>\0<time>\0<acctNumber>\0<exchange>\0<side>\0<shares>\0<price>\0<cumQty>\0<avgPrice>\0

    execId: генерируйте уникальную строку, например exec12345.
    cumQty: накопленное количество (если исполнение частичное).


4. Статус счета и Позиции

MC постоянно запрашивает обновления, чтобы рисовать вкладку "Trade Manager".
Обновление счета (Код 6):
6\0<version>\0<tag>\0<value>\0<currency>\0<accountName>\0

    Шлите теги: NetLiquidation, TotalCashValue, BuyingPower. Без них MC решит, что денег 0 и не даст торговать.

Позиции (Код 61):
Если в эмуляторе уже есть открытые позы:
61\0<version>\0<account>\0<ContractBlock>\0<position_size>\0<avgCost>\0
Когда позиции кончаются, шлите: 62\0<version>\0 (Position End).
5. Техническая реализация "Бинарщины"
Поскольку вы работаете с версией 100+ (V100), запомните алгоритм упаковки:

    Сформируйте все поля в одну строку, разделяя их \0.
    Пример: "15\02\01\0U12345\0" (длина 12 байт).
    Переведите длину (12) в 4 байта (Integer Big-Endian). В Hex это 00 00 00 0C.
    Склейте: [00 00 00 0C] + [31 35 00 32 00 31 00 55 31 32 33 34 35 00].
    Отправьте в TCP-сокет.

Ошибки связи (Heartbeats):
Если данных нет, раз в 1-2 секунды шлите системное время (Код 49):
49\01\0<timestamp>\0 — это предотвратит разрыв соединения со стороны MultiCharts.
С чего начать кодинг:
Реализуйте сначала Код 4 (Error) с кодами 2104, 2106 и 2158. Пока MC их не получит, он будет писать "Connecting", даже если TCP сессия открыта.

Продолжаем детализацию. Чтобы MultiCharts (MC) не просто «видел» соединение, а мог совершать сделки и подписываться на данные, нужно реализовать блок Place Order (Код 3), Contract Details (Код 10) и систему уровней котировок.
1. Структура Place Order (Код 3) — «Самый длинный пакет»
Когда MC отправляет приказ, он шлет более 40 полей. Эмулятор должен прочитать их все, даже если не использует, чтобы не «сломать» очередь байтов.
Порядок ключевых полей в пакете Код 3 (V100+):

    ID сообщения (3)
    orderId (ID, который вы выдали в NextValidId)
    Блок контракта (см. предыдущий ответ: conId, symbol, и т.д.)
    secIdType / secId (обычно пустые)
    action (BUY / SELL / SSHORT)
    totalQuantity (число)
    orderType (MKT / LMT / STP / STP LMT / TRAIL)
    lmtPrice (цена для лимиток)
    auxPrice (цена для стопов или шаг трейлинга)
    tif (Time In Force: DAY / GTC / IOC)
    ocaGroup (для связанных ордеров)
    account (номер счета, например U12345)
    openClose (O / C)
    origin (0 — клиент, 1 — фирма)

Технический секрет: "Версионность полей"

6. Секреты стабильности TCP-потока (Low Level)
Для эмулятора на языках типа C# или C++:

    Nagle's Algorithm: Обязательно отключите алгоритм Нейгла в настройках сокета (TCP_NODELAY). Иначе мелкие пакеты котировок будут группироваться, и в MC возникнут задержки (лагающие котировки).
    Дробление пакетов: Иногда одно сообщение (например, история за год) не влезает в один TCP-пакет (MTU ~1500 байт). Ваш парсер на стороне эмулятора должен уметь «дособирать» сообщение, пока не вычитает количество байт, указанное в 4-байтовом префиксе V100.
    Big-Endian: Помните, что длина сообщения в заголовке — это Signed Int32 в формате Big-Endian.

1. Сложные типы ордеров (Trailing Stop и Bracket)

Как писать бинарно (финальный лайфхак):

Итоговая архитектура вашего сервера (Simplified):

    Слой TCP: Принимает байты, парсит заголовок (4 байта длины).
    Диспетчер (Router): Смотрит на первый байт после длины (ID сообщения) и вызывает нужный обработчик.
    Обработчик (Handler): Считывает остальные поля по списку \0.
    Генератор (Emitter): Формирует ответ, упаковывает его в V100 и отправляет обратно.

1. Специфика Фьючерсов (SEC_TYPE = FUT)
Для MultiCharts фьючерсы — это основной инструмент. Чтобы график открылся, эмулятор должен правильно обработать «даты».

    Поля контракта:
        lastTradeDateOrContractMonth: Формат YYYYMMDD (например, 20240621). Если прислать только месяц, MC может не принять контракт.
        multiplier: Обязательное поле (строка). Для Gold (GC) — 100, для Euro (6E) — 125000. Если прислать 0, расчет прибыли в MC будет нулевым.
    Докачка (Historical Data): При запросе исторических данных для фьючерсов, MC часто просит whatToShow = TRADES. Если прислать MIDPOINT, данные могут не отобразиться на свечном графике.

2. Управление «ID следующего ордера»
Это главная точка синхронизации.

    При каждом подключении клиента сервер обязан отправить сообщение тип 9 (NextValidId).
    Если эмулятор перезагрузился, он должен начинать orderId с числа большего, чем было в прошлый раз. Если MC попытается отправить ордер с ID, который уже был использован, возникнет критическая ошибка, и ордер будет отклонен.

Чтобы в MultiCharts вкладка "Trade Manager" была живой:

    Поля: execId, commission (double), currency, realizedPnL.
    Код 13 (Account Update): Шлите периодически (раз в 5 сек).
    Важные теги:
        UnrealizedPnL: текущая плавающая прибыль.
        RealizedPnL: зафиксированная прибыль.
        EquityWithLoanValue: общая стоимость счета.

5. Принципы бинарной упаковки (V100+) — Шпаргалка
Для любой функции отправки:

    Разделитель: Каждое поле заканчивается \0 (0x00).

6. Порядок обработки сообщений (Workflow)
Для стабильной работы эмулятор должен работать как бесконечный цикл:

    Ждать API\0 (Handshake).
    Ждать START_API (Код 71).
    Выдать NextValidId (Код 9) и ManagedAccounts (Код 15).
    Выдать Системные ошибки 2104, 2106 (Код 4), чтобы "зажечь" зеленый индикатор связи в MC.
    Слушать запросы:
        Код 20 -> выдать историю (Код 16) + FINISHED.
        Код 1 -> запустить поток тиков (Код 2).
        Код 3 -> создать ордер, ответить Submitted (Код 11) и Execution (Код 5).

7. Почему эмулятор может "лагать" в MC?

    Забытый \0: Если пропустить разделитель, MC "склеит" два поля и выдаст ошибку "Invalid field".
    Неверная длина: Если в 4-байтовом заголовке указана длина меньше, чем реально в пакете, MC прочитает только часть, а остаток байтов "отравит" следующий пакет.
    Таймауты: Если не слать Heartbeat (Код 49 - системное время) хотя бы раз в минуту, MC разорвет соединение.


1. Обработка рыночных сессий (Trading Hours)
MultiCharts запрашивает время работы биржи, чтобы знать, когда рисовать графики.

    Отмена подписки (Код 2): Клиент присылает 2\0<version>\0<tickerId>\0. После этого эмулятор должен прекратить слать тики для этого ID. Это критично для экономии трафика.

3. Тики «размера» и «цены» (Код 2 и 4)
В IB API есть два способа передачи котировок:

    Tick Price (Код 2): Передает цену (Bid, Ask, Last).
    Tick Size (Код 4): Передает объем (Bid Size, Ask Size, Last Size).
    Важно: Для корректного отображения стакана и ленты сделок в MC, вы должны слать их в паре. Например, если изменился Bid, шлите сначала цену (Код 2, tickType=1), а затем объем (Код 4, tickType=0).

4. Тикерные ошибки (Tick Snapshot Error)
Если MC запрашивает тикер, которого у вас нет в базе:

    Шлите Код 4 (Error) с id = tickerId и кодом ошибки 200 (No security definition has been found). Это заставит MC прекратить попытки подписки.

5. Поля для управления позициями (Positions Multi-Account)
Если вы эмулируете несколько счетов:

    Код 61 (Position): Передает текущую открытую позицию.
    Код 62 (Position End): Обязательный маркер после передачи всех позиций. Если его не прислать, MC будет бесконечно «крутить» колесико загрузки портфеля.

6. Секретные системные запросы (Keep Alive)

    Код 49 (Current Time): Раз в 30-60 секунд эмулятор должен отправлять UNIX-timestamp.
    49\01\0<timestamp>\0. Это единственный способ подтвердить MC, что сервер не завис, если на рынке нет активных торгов.

7. Специфика сообщений для Опционов (Volatility)
Если ваша цель — опционы, MC будет слать запрос на расчет волатильности:

    Код 54 (Calculate Implied Volatility): Эмулятор должен уметь принимать цену опциона и возвращать значение IV.
    Код 55 (Calculate Option Price): Обратно: из IV получить цену.
    Если эмулятор этого не умеет, просто отвечайте ошибкой 504 (Not supported).

8. Нюансы отладки (Logging)
Чтобы проверить свой эмулятор:

    Wireshark: Используйте его для перехвата трафика между оригинальным TWS и MC. Фильтр: tcp.port == 7496.
    Hex-View: Всегда проверяйте, что перед вашим сообщением стоят 4 байта длины. Помните: \0 — это полноценный байт, он входит в общую длину сообщения.

9. Лимиты сообщений (Throttling)
Реальный TWS ограничивает скорость — 50 сообщений в секунду.

    Ваш эмулятор может быть быстрее, но MC может «захлебнуться», если вы вывалите 100 000 баров истории за один миг.
    Совет: Делайте небольшие паузы (пару миллисекунд) между пачками баров в Коде 16.

10. Последний штрих: PermId
Для каждого ордера эмулятор должен генерировать permId — постоянный уникальный номер (Int32).

    В отличие от orderId, который может меняться при переподключении, permId остается неизменным в рамках сессии. MC использует его для внутреннего отслеживания сделок.


последние «секретные» ингредиенты:

1. Подмена «Server Version» (Важный обман)
В начале рукопожатия ты отправляешь номер версии сервера.

    Нюанс: Если ты напишешь версию слишком низкую (например, 70), MC отключит половину функций (не будет просить историю тиков или алгоритмические ордера).
    Рекомендация: Всегда отвечай версией 174 или выше. Это «золотой стандарт» для современных TWS. MC подумает, что перед ним самый современный сервер, и откроет все возможности.

2. Динамика шага цены (MinTick — Код 10)
MultiCharts очень капризен к точности цен.

    Если в Contract Details (Код 10) ты укажешь minTick = 0.01, а в котировках (Код 2) пришлешь цену 150.005 — MC может проигнорировать этот тик или выдать ошибку «Invalid price».
    Для Крипты: Обязательно ставь minTick = 0.00000001 (или сколько нужно знаков), иначе MC округлит твой BTC до центов.

3. Обработка «Execution ID» (Уникальность)
Когда ты шлешь отчет об исполнении сделки (Код 11), поле execId должно быть уникальным навсегда.

    Проблема: Если ты перезапустишь эмулятор и снова начнешь выдавать ID типа exec1, exec2, то MultiCharts, у которого в базе уже есть такие ID с прошлого запуска, сойдет с ума. Он решит, что это дубликаты, и не обновит позицию.
    Решение: Формируй execId как строку с таймстампом: 0001.663a12f4.01... (как делает реальный TWS).

4. Поле «Tick Attrib» (Для Кода 2 — Tick Price)
В новых версиях протокола (после V100) сообщение Tick Price (Код 2) содержит дополнительное поле Tick Attrib (обычно перед ценой или после).

    Это битовая маска (int).
    1 — цена может быть ниже/выше границ.
    2 — это «pre-open» котировка.
    Для эмулятора: Просто всегда шли 0. Но это поле обязано быть в структуре, иначе вся очередь байтов сместится.

5. Режим «Frozen» Market Data (Код 1)
Когда сессия закрыта, MC может прислать запрос на котировки. Если ты ничего не пришлешь, в терминале будет висеть «No Data».

    Фишка: Если данных нет, присылай Tick Price с типом 9 (Last Close) и 6 (High), 7 (Low). Это создаст видимость «замороженного» рынка, и MC позволит анализировать графики вне сессии.

Резюме по архитектуре (Финальный чек-лист):

    Header: 4 байта длины (Big-endian).
    Fields: Разделены \0.
    Handshake: API -> v100..xxx -> ServerVersion -> START_API.
    KeepAlive: Код 49 раз в 30 сек.
    Errors: Обязательно коды 2104, 2106 для статуса «Connected».
    Contracts: conId должен быть стабильным (не меняться для одного тикера).

Технический совет: 
Если будешь писать на C#, используй BinaryWriter с перевернутым порядком байтов.


1. Досборка полей в Order Status (Код 11)
В файле часто зашиты поля, которые эмулятор обязан отдавать в ответ на приказ, чтобы MC не выдал «Order Exception»:

    PermId: В бинарном потоке это поле идет сразу после avgFillPrice. Это 4-байтовое целое, которое MC сохраняет как уникальный ключ.
    ParentId: Если ты эмулируешь сложный ордер, это поле должно содержать orderId основного ордера, иначе связка (bracket) развалится.
    LastFillPrice: MC ждет цену последнего исполнения именно в этом сообщении, а не только в отчете об исполнении (Execution Report).

2. Специфика «Market Data Type» (Код 68)

    Если эмулятор не ответит на запрос 68, MC может бесконечно ждать подтверждения, что данные «Real-Time».
    Бинарный ответ: 68\01\01\0 (где первая 1 — версия, вторая 1 — код Live-данных).

3. Блок ContractDetails — «Хитрое поле»
Проверь, есть ли в твоем понимании кода поле callable и putable для облигаций или underConId для опционов. Если файл содержит эти структуры, то при эмуляции обычных акций ты обязан слать там пустые строки (\0), иначе MC решит, что пакет не до конца прочитан, и «сдвинет» следующее сообщение.
4. Tick News и Tick Generic
В файле часто прописаны genericTickList. Это когда MC просит не просто цену, а, например, «Volume Rate» или «Option Historical Volatility».

    Если ты видишь в коде обработку строки с кодами (например, 100,101,165), твой эмулятор должен уметь игнорировать эти запросы, отправляя в Tick Price стандартные значения, иначе график будет «пустым».

5. NextValidId — критический момент
Файл указывает, что без сообщения тип 9 кнопка «Place Order» в MultiCharts останется серой.

    Бинарно: 9\01\0<ID>\0.


    Handshake (API -> версии -> ServerTime).
    Managed Accounts (Код 15).
    NextValidId (Код 9).
    Historical Data (Код 16 + FINISHED).
    Market Data (Коды 2, 4).
    Order Placement/Status (Коды 3, 11, 5).



данные для полной эмуляции TWS, включая систему «Умных уведомлений» (Код 12), детальные отчеты по комиссиям (Код 8), настройки контрактов, Market Scanner и коды ошибок для активации соединения. Для корректной работы требуются сообщения типа «Market data farm connection is OK» (2104, 2106, 2158) и строгий учет разделителей \0 и 
big-endian порядка байтов.



Все данные передаются в бинарном виде: [4 байта длины (Big-Endian)] + [Тело сообщения]. 

Каждое поле внутри тела заканчивается нулевым байтом \0.

Часть 1: Соединение и инициализация (Handshake)
Это критическая фаза. Без точного соблюдения последовательности MultiCharts (MC) сбросит TCP-сессию.

    Начало (Клиент > Эмулятор):
    MC отправляет строку API\0. Сразу за ней идет диапазон версий: v100..174\0 (или аналогичный).
    Ответ (Эмулятор > Клиент):
    Эмулятор обязан ответить версией сервера и временем.
        Формат: <server_version>\0<connection_time>\0
        Пример: 174\020231201 12:00:00 UTC\0 (Используйте версию не ниже 150 для поддержки всех функций).
    Подтверждение (Клиент > Эмулятор):
    MC присылает сообщение START_API (код 71).
        Структура: 71\0<version>\0<clientId>\0<optionalCapabilities>\0
    Финальная настройка (Эмулятор > Клиент):
    После START_API эмулятор обязан отправить два сообщения, иначе MC не активирует торговые кнопки:
        Managed Accounts (Код 15): Список доступных счетов.
        15\01\0<account_list>\0 (например, U12345\0).
        Next Valid ID (Код 9): Следующий свободный ID для ордера.
        9\01\0<orderId>\0 (например, 1\0).
        Системные статусы (Код 4): Чтобы загорелся «зеленый свет» связи.
        Отправьте коды ошибок 2104 (Market data farm OK) и 2106 (HMDS data farm OK).


Часть 2: Закачка данных (Historical Data)
Когда вы открываете график, MC запрашивает исторические бары.

    Запрос (Клиент > Эмулятор - Код 20):
    Пакет содержит: 20\0<reqId>\0<ContractBlock>\0<endDateTime>\0<barSizeSetting>\0<durationStr>\0<useRTH>\0<whatToShow>\0<formatDate>\0<keepUpToDate>\0<chartOptions>\0
        Важно: whatToShow определяет тип данных (TRADES, BID, ASK, MIDPOINT).
    Ответ (Эмулятор > Клиент - Код 16):
    Сначала идет заголовок: 16\0<reqId>\0<start_date>\0<end_date>\0<bar_count>\0.
    Затем — массив баров. Каждый бар:
    <date>\0<open>\0<high>\0<low>\0<close>\0<volume>\0<barCount>\0<WAP>\0
    Маркер завершения:
    После последнего бара эмулятор шлет то же сообщение 16, но с текстом FINISHED вместо данных бара. Это сигнал для MC, что загрузка окончена.


Часть 3: Докачка и Real-time данные (Streaming)
После загрузки истории MC подписывается на «живые» котировки для обновления графиков.

    Подписка (Клиент > Эмулятор - Код 1):
    1\0<tickerId>\0<ContractBlock>\0<genericTickList>\0<snapshot>\0<regulatorySnapshot>\0<mktDataOptions>\0
    Поток котировок:
    Эмулятор должен слать два типа сообщений:
        Tick Price (Код 2): 2\01\0<tickerId>\0<tickType>\0<price>\0<attribs>\0
        Типы: 1=Bid, 2=Ask, 4=Last, 6=High, 7=Low, 9=Close.
        Tick Size (Код 4): 4\01\0<tickerId>\0<tickType>\0<size>\0
        Типы: 0=Bid Size, 3=Ask Size, 5=Last Size, 8=Volume.
    Обновление баров (Real-time Bars - Код 50):
    Если MC просит 5-секундные бары:
    50\0<reqId>\0<time>\0<open>\0<high>\0<low>\0<close>\0<volume>\0<wap>\0<count>\0


Часть 4: Торговые операции (Orders & Execution)
Самый сложный бинарный блок. Ошибки здесь приводят к "зависанию" ордеров.

    Размещение (Клиент > Эмулятор - Код 3):
    MC шлет огромный пакет Place Order. Эмулятор должен извлечь orderId, action (BUY/SELL), totalQuantity, orderType и lmtPrice.
    Подтверждение (Эмулятор > Клиент - Код 11):
    Сразу после получения ордера нужно отправить его статус:
    11\0<orderId>\0Submitted\0<filled>\0<remaining>\0<avgFillPrice>\0<permId>\0<parentId>\0<lastFillPrice>\0<clientId>\0<whyHeld>\0
        PermId: Сгенерируйте случайное число, оно нужно MC для идентификации.
    Исполнение (Execution Report - Код 5):
    Чтобы позиция открылась в MC:
    5\0<reqId>\0<orderId>\0<ContractBlock>\0<execId>\0<time>\0<acctNumber>\0<exchange>\0<side>\0<shares>\0<price>\0...
        execId должен быть уникальным для каждой сделки.
    Статус счета (Код 6):
    Чтобы MC знал баланс: 6\02\0<tag>\0<value>\0<currency>\0<account>\0 (Теги: CashBalance, NetLiquidation).
    Отмена (Код 4):
    Если клиент шлет отмену, эмулятор отвечает кодом 11 со статусом Cancelled и сообщением об ошибке 4 (код ошибки 202).

Совет для эмулятора: Весь бинарный обмен должен строго соблюдать порядок полей  
Если вы пропустите один \0 (например, в блоке ContractBlock), все последующие данные в MC "съедут" и программа выдаст ошибку парсинга.
